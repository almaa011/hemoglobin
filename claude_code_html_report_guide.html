1 Overview

What Claude Code is and how it works

Understanding the architecture matters for report generation: Claude Code is not just a chat window — it has a persistent memory layer, isolated subagents, and a context-budget constraint that determines everything.

Claude Code is an AI coding assistant that reads entire codebases, edits files, and runs shell commands. It is available as a CLI, IDE extension (VS Code, JetBrains), desktop app, and web interface. The core loop is: read context → plan → execute → verify.

CLAUDE.md — persistent project instructions

A CLAUDE.md file at the project root gives Claude persistent instructions for coding standards, workflow rules, and architectural decisions. Claude reads it at the start of every session. Target under 200 lines per file — larger files consume context budget and reduce instruction adherence. Rules can be scoped to specific file patterns using YAML frontmatter in .claude/rules/ subdirectories.

Memory system

Claude Code stores accumulated learnings in ~/.claude/projects/<project>/memory/. The first 200 lines of MEMORY.md (or 25 KB) load at session start. Detailed topic files are lazy-loaded on demand. This keeps startup context cost low while making prior learnings available.

Subagents and parallelism

Subagents are isolated agent instances that handle focused subtasks. Each runs in a separate context window — intermediate tool calls stay inside; only the final message returns to the parent. Multiple subagents can run concurrently: independent tasks finish in the time of the slowest one, not the sum. Subagents can have restricted tool access (read-only, write-only, etc.) via the tools field.

Key insight for report generation

The context window is the primary constraint. Every file Claude reads, every command output, every conversation turn consumes context. Subagents offload exploratory work (reading dozens of source files) to separate context windows — the parent gets a summary, not the raw content. Design your workflows to exploit this: use subagents for research, the main agent for writing.

Verification signals

Claude stops when work "looks done." Without a runnable check (test suite, build command, linter), human review becomes the only error-detection loop. For report generation, this means: give Claude a validator that produces pass/fail — an HTML validator, a link checker, or a screenshot comparison — and the loop closes itself.

2 Stack selection

The right stack for HTML reports

Three viable approaches exist. Each has a different tradeoff between data integration ease, chart interactivity, and portability. Choose based on whether the report is generated once or reused.

Stack Self-containment Chart quality Data integration Build complexity Best for
Pure HTML + JS Excellent Good (manual) Manual / JSON inline None Portability-first, one-file delivery
Python + Jinja2 Good Excellent (matplotlib) Native (pandas, etc.) Low Data pipelines, programmatic generation
Python + Plotly Large (+3 MB) Excellent (interactive) Native Low Interactive dashboards with filter controls
Node.js + templating Fair Excellent Good Medium Complex UI with React/Vue components

Recommendation: Pure HTML/JS for maximum portability

For reports that need to open offline, be emailed as a single attachment, or work in restricted environments — pure HTML/JS wins. Embed data as a JSON literal in a <script> block, render charts via Canvas API or a bundled charting library, and keep all CSS inline. The entire deliverable is one .html file that opens in any browser from 2018 onward.

When to use Python + Jinja2 instead

If the report is generated from a data pipeline (pandas DataFrames, database queries, numpy arrays), Python wins on data integration. Use Jinja2 templates to separate the HTML structure from the Python logic. Use matplotlib with savefig(BytesIO) + base64 encoding to embed static charts with no JavaScript required. This approach produces smaller files than Plotly-embedded HTML.

Avoid this combination

Node.js + React for a one-off technical report. The build toolchain complexity (webpack, babel, npm packages) vastly exceeds the value. Reserve React for reports that need state management, filtering, or real-time data. For static technical reports, it is pure overhead.

Chart.js — recommended for self-contained HTML

When to use it: Bar, line, pie, radar, doughnut charts in a self-contained HTML file. 92 KB gzipped. Canvas-based rendering. Tree-shakeable since v3 — import only the chart types you use to shave ~56 KB from the bundle.

Self-contained embed: Download chart.umd.min.js from the CDN once, paste its content into a <script> block. No npm, no build step.

📄 minimal-chart.html — Chart.js inline bar chart
<canvas id="myChart" width="600" height="300"></canvas>
<script>
// Chart.js UMD bundle pasted here (chart.umd.min.js content)
const ctx = document.getElementById('myChart').getContext('2d');
new Chart(ctx, {
  type: 'bar',
  data: {
    labels: ['Red', 'Blue', 'Yellow', 'Green'],
    datasets: [{
      label: 'Votes (n=120)',       // ← always include a meaningful label
      data: [42, 31, 27, 20],
      backgroundColor: ['#4f46e5', '#0891b2', '#d97706', '#059669']
    }]
  },
  options: {
    responsive: true,
    layout: { padding: 12 },        // ← prevents label clipping
    plugins: {
      title: { display: true, text: 'Color preference survey' },
      legend: { position: 'top' }
    },
    scales: {
      x: { title: { display: true, text: 'Color' } },   // ← axis labels
      y: { title: { display: true, text: 'Votes' }, beginAtZero: true }
    }
  }
});
</script>

► layout.padding prevents data labels from being clipped at the canvas edge — a top-5 Chart.js bug.

Plotly.js — for interactive or Python-generated reports

When to use it: When your report needs zoom, pan, hover tooltips, or is generated from Python. The Python API (fig.to_html(include_plotlyjs='include')) produces a fully offline HTML file but adds ~3 MB inline. For JavaScript, use the Basic partial bundle (365 KB gzipped) when you only need scatter/bar/pie.

🐍 generate_report.py — matplotlib → base64 PNG embed
import matplotlib.pyplot as plt
import io, base64

fig, ax = plt.subplots(figsize=(8, 4), dpi=120)
ax.bar(['A', 'B', 'C'], [42, 31, 27], color='#4f46e5')
ax.set_title('Category counts')          # ← title is mandatory
ax.set_xlabel('Category')               # ← label both axes
ax.set_ylabel('Count')
ax.set_ylim(0, 50)                       # ← explicit range prevents odd scaling
fig.tight_layout()                       # ← prevents clipped tick labels

buf = io.BytesIO()
fig.savefig(buf, format='png')
buf.seek(0)
img_b64 = base64.b64encode(buf.getvalue()).decode('utf-8')

html_snippet = f'<img src="data:image/png;base64,{img_b64}" alt="Category counts bar chart">'
# ↑ always include a descriptive alt attribute

► fig.tight_layout() is critical — without it, axis labels and titles clip on the figure boundary.

Common chart failure patterns

new Chart(ctx, {

  data: {
    labels: ['A', 'B', 'C'],

✓ Good — title, labels, legend

  type: 'bar',
  data: {
    labels: ['A', 'B', 'C'],
    datasets: [{
      label: 'Frequency',
      data: [10, 20, 30]
    }]
  },
  options: {
    plugins: { title: {
      display: true,
      text: 'Letter frequency'
    }},
    scales: {
      x: {title:{display:true,text:'Letter'}},

    }
  }

Color-only encoding

Never use color as the only differentiator between data series. ~8% of men have red-green color blindness. Always pair color with shape, pattern, or direct labels. In Chart.js, add pointStyle variations to scatter charts and use datalabels plugin to label bars directly rather than relying on legend color matching alone.

4 Self-Contained HTML

How to make an HTML file truly self-contained

A self-contained HTML file opens correctly with no internet connection, no local server, and no sibling files. Every resource — fonts, CSS, JS, images — must live inside the single .html file.

Fonts: use system font stacks (preferred)

The simplest approach: use a system font stack that requires zero embedding and works on every platform without a single byte of overhead.

CSS — system font stack (zero external dependency)
body {
  font-family: -apple-system, BlinkMacSystemFont, 'Segoe UI',
               Roboto, 'Helvetica Neue', Arial, sans-serif;
}
code, pre {
  font-family: 'JetBrains Mono', 'Cascadia Code', 'Fira Code',
               'SF Mono', Consolas, 'Courier New', monospace;
}

Fonts: base64 embed (when a specific typeface is required)

Base64-encoded fonts add ~33% size overhead versus the original file and block CSS parsing. The break-even point is approximately 10 KB: below this threshold, eliminating one HTTP round-trip outweighs the payload increase. For WOFF2 files (typically 15–80 KB), this makes external loading + font-display: swap the better choice unless offline use is mandatory.

CSS — base64 font embed pattern
@font-face {
  font-family: 'MyFont';
  src: url('data:font/woff2;base64,d09GMgABAAAAA...') format('woff2');
  font-weight: 400;
  font-style: normal;
  font-display: block; /* block instead of swap — file is already inline */
}
/* Generate the base64 string: */
/* python: base64.b64encode(open('font.woff2','rb').read()).decode() */

CSS: always inline in <style>

Never use @import in a self-contained file. @import creates a serial dependency chain: the browser must finish downloading and parsing the first stylesheet before it can even request the imported one. Inline all CSS in one <style> block in <head>.

JavaScript libraries: paste the minified source

For Chart.js or similar: download the UMD minified bundle once, paste its content inside a <script> tag at the bottom of <body>. Do not use a CDN URL. CDN links require internet and may return 404 after the CDN provider changes their URL structure.

HTML structure — correct load order for self-contained reports
<!DOCTYPE html>
<html lang="en">          <!-- lang= is mandatory for screen readers -->
<head>
<meta charset="UTF-8">   <!-- must appear within first 1024 bytes -->
<meta name="viewport" content="width=device-width, initial-scale=1">
<meta name="color-scheme" content="light dark">
<title>Report Title</title>
<style>
  /* ALL CSS here — no @import, no external href */
  :root { --accent: #4f46e5; }
  @media (prefers-color-scheme: dark) {
    :root { --accent: #818cf8; }
  }
</style>
</head>
<body>
  <!-- Content -->
<script>
  /* charting library source pasted here */
  /* page-specific chart init code here  */
</script>
</body>
</html>

Images: SVG inline, raster as base64

For icons and diagrams, use inline <svg> — it stays structured text, compresses well with gzip, and can be styled with CSS. For raster images (matplotlib output, screenshots), embed as data:image/png;base64,... in an <img src=""> attribute.

Dark mode: CSS custom properties + media query

Define your entire color palette as CSS custom properties at :root level, then redefine them inside @media (prefers-color-scheme: dark). Every element that uses var(--token) automatically switches theme. Never hardcode hex colors outside the token block.

CSS — dark mode via custom properties (single-source-of-truth)
:root {
  --bg:     #f9f8f6;
  --text:   #1c1a18;
  --accent: #4f46e5;
}
@media (prefers-color-scheme: dark) {
  :root {
    --bg:     #0f1117;
    --text:   #e8e4dc;
    --accent: #818cf8;   /* lighter shade for dark bg contrast */
  }
}
body { background: var(--bg); color: var(--text); }
a    { color: var(--accent); }
/* Never write: background: #f9f8f6; outside :root */

5 CLAUDE.md

CLAUDE.md instructions for HTML report quality

Paste this block into your project's CLAUDE.md. These rules enforce the most common failure modes that AI agents commit when generating HTML reports. Rules are written as testable statements, not aspirations.

📄 CLAUDE.md — HTML report quality rules (ready to paste)
## HTML Report Rules

### Output requirements
- Output is a single .html file with zero external dependencies. No CDN links, no
  relative asset paths, no external CSS files. The file must open correctly offline.
- Every chart must have: a title, labeled x-axis, labeled y-axis, and a legend or
  direct data labels. A chart without all four is a blocking defect.
- No placeholder data (Lorem ipsum, [VALUE], TODO, sample data, dummy). Every data
  point in every chart must come from the actual source data for this report.
- All colors are defined as CSS custom properties at :root level. Never hardcode
  a hex color outside the :root block, except inside JS chart configuration where
  CSS vars are unavailable — in that case, add a comment mapping it to the token.
- The report must render correctly in both light and dark system themes. Test by
  defining @media (prefers-color-scheme: dark) with a full :root override.
- No inline style attributes that override layout (width, margin, padding on
  structural elements). All layout lives in the <style> block.
- Responsive at 1200px and 768px viewport widths. Tables must be wrapped in
  overflow-x: auto containers. No fixed-pixel widths on content columns.

### Chart requirements
- Chart.js: always set layout.padding to at least 12 to prevent label clipping.
  Always set beginAtZero: true on numeric axes unless the data range argues against it.
- matplotlib: always call fig.tight_layout() before savefig(). Set dpi=120 minimum.
  Always provide alt text in the <img> tag describing the chart content.
- Do not use color as the only distinguishing feature between data series.

### HTML structure requirements
- <html lang="en"> — lang attribute is mandatory.
- <meta charset="UTF-8"> must be within first 1024 bytes of document.
- <meta name="viewport" content="width=device-width, initial-scale=1"> required.
- <meta name="color-scheme" content="light dark"> required when dark mode is supported.
- All JavaScript is in one <script> block at the bottom of <body>.
- No duplicate id attributes anywhere in the document.

### Verification checklist (run before declaring done)
- [ ] Every chart has title, axis labels, legend/data labels
- [ ] File opens offline in a browser (no network requests)
- [ ] Color palette works in both light and dark mode
- [ ] No placeholder or sample data
- [ ] No fixed-pixel widths that break at 768px
- [ ] No duplicate IDs
- [ ] lang, charset, viewport, color-scheme meta tags present

Why testable rules beat aspirational ones

"Format code properly" cannot be verified. "Use 2-space indentation" can be checked by a linter. The rules above are written so that each one has a clear pass/fail: you can grep for missing axis labels, open DevTools Network tab to confirm zero external requests, and run the W3C validator for duplicate IDs. Testable rules close the verification loop without human review.

6 Failure modes

Ten specific failure modes — and how to fix each

These are the failure modes observed empirically in AI-generated HTML reports. Each has a structural cause, not just a surface symptom, and a specific fix that closes the issue.

1
Missing chart title, axis labels, or legend

The AI generates functional chart code but omits all metadata. The chart renders as a blob of colored bars with no context. A reader cannot determine what the chart measures, what units the axes use, or what the series represent.

Fix: add title, x-axis label, y-axis label, and either a legend or direct data labels as mandatory fields. Make absence of any of these a blocking rule in CLAUDE.md.

2
Data label clipping at canvas edge

Chart.js renders labels that extend beyond the canvas boundary. The browser clips them. Root cause: layout.padding defaults to 0, so labels at the top of tall bars are cut off. Also occurs with rotated x-axis tick labels.

Fix: set layout: { padding: 12 } in Chart.js options. For rotated labels, set ticks: { maxRotation: 45 } and increase bottom padding.

3
Hardcoded hex colors outside the token system

Colors scattered throughout the stylesheet as literal hex values. The report looks fine in light mode but has broken contrast in dark mode. Changing the accent color requires a find-and-replace across dozens of locations, missing some.

Fix: define a CSS custom property for every semantic color at :root level. Reference only var(--token) in the rest of the stylesheet. One change propagates everywhere.

4
External CDN dependencies

Report uses <script src="https://cdn.jsdelivr.net/..."> or @import url('https://fonts.googleapis.com/...'). The report appears to work during generation (when the machine has internet) but fails silently offline. CDN URLs also rot over time.

Fix: download any required library as a local file or paste its minified content inline in a <script> block. Use system font stacks to eliminate font CDN dependencies entirely.

5
No dark mode support

The report is designed for a white background. On macOS, Windows 11, or iOS in dark mode, the browser renders the white background as white — a blinding white page. Embedded chart images with white backgrounds become jarring white rectangles.

Fix: add <meta name="color-scheme" content="light dark"> and a complete @media (prefers-color-scheme: dark) :root override. For matplotlib images, set plt.style.use('dark_background') or check the user's preferred scheme server-side.

6
Placeholder or sample data

The AI generates a structurally correct chart populated with [12, 34, 56] or ['Category A', 'Category B'] instead of the actual data from the source. The page looks complete during code review but conveys no information.

Fix: in CLAUDE.md, make placeholder data a blocking defect. In the prompt, explicitly provide the data or a reference to where it lives. The AI must use the provided data, not invented values.

7
Fixed-pixel widths that break on mobile

Report uses width: 800px on charts or content columns. At 768px viewport, content overflows. Tables are especially problematic — they overflow horizontally without a scroll container, breaking the page layout.

Fix: use max-width + width: 100% on all content. Wrap every <table> in <div style="overflow-x: auto">. Set Chart.js responsive: true. Canvas elements automatically adapt.

8
Color-only data encoding

The chart distinguishes data series by color alone. ~8% of men and ~0.5% of women have red-green color blindness. A pie chart with red and green slices is unreadable for a significant portion of readers. Legends that say "Red line = Series A, Green line = Series B" are inaccessible.

Fix: pair color with shape (different pointStyle in Chart.js), pattern fills, or direct data labels on chart elements. Test by simulating deuteranopia using browser devtools accessibility filters.

9
Missing lang, charset, or viewport meta tags

22.3% of the top 1 million websites omit the lang attribute. Without it, screen readers apply incorrect pronunciation rules. Without charset=UTF-8 in the first 1024 bytes, the browser may misparse the document. Without viewport, mobile browsers default to a 980px desktop viewport and scale the page down, making text unreadably small.

Fix: the required trio — <html lang="en">, <meta charset="UTF-8"> (first in head), <meta name="viewport" content="width=device-width, initial-scale=1"> — is boilerplate. Add to every CLAUDE.md template.

10
Non-deterministic output across identical prompts

Two invocations of the same prompt produce reports with different color palettes, different label placements, and different data ordering. The AI samples from its distribution each time. This makes the report feel unreviewed and inconsistent when compared to a prior version.

Fix: define the visual system completely in CLAUDE.md (color tokens, font stack, component structure) so there is nothing left for the AI to invent. Lower temperature in API calls if using the API directly. The less design latitude given, the more reproducible the output.

Self-contained HTML report example · Canvas 2D API · no external dependencies

Hover or resize — chart adapts to container width. Axes, title, and legend are all computed in JS.

The complete code for that chart (data, layout, rendering) fits in ~70 lines of vanilla JavaScript with no dependencies:

📄 report-example.html — ~70-line self-contained chart
<!DOCTYPE html>
<html lang="en">
<head>
<meta charset="UTF-8">
<meta name="viewport" content="width=device-width, initial-scale=1">
<meta name="color-scheme" content="light dark">
<title>Bundle Size Report</title>
<style>
:root { --bg: #f9f8f6; --text: #1c1a18; --border: #e2ddd8; }
@media (prefers-color-scheme: dark) {
  :root { --bg: #0f1117; --text: #e8e4dc; --border: #2d3748; }
}
body { margin: 0; background: var(--bg); color: var(--text);
       font-family: system-ui, sans-serif; padding: 32px; }
h1 { font-size: 1.4rem; margin: 0 0 4px; }
p.sub { margin: 0 0 20px; font-size: 13px; color: #888; }
canvas { display: block; width: 100%; max-width: 720px; }
</style>
</head>
<body>
<h1>JS Charting Libraries — Gzipped Bundle Size</h1>
<p class="sub">Smaller = better for self-contained HTML delivery</p>
<canvas id="c" aria-label="Bar chart: library bundle sizes" role="img"></canvas>
<script>
const data = [
  { label: 'D3 (custom)',      kb: 15,   color: '#059669' },
  { label: 'Chart.js',         kb: 92,   color: '#4f46e5' },
  { label: 'ECharts',          kb: 167,  color: '#0891b2' },
  { label: 'Plotly Basic',     kb: 365,  color: '#d97706' },
  { label: 'Plotly Full',      kb: 1400, color: '#dc2626' },
];
const pad = { t: 40, r: 20, b: 60, l: 120 };
function draw() {
  const canvas = document.getElementById('c');
  const W = canvas.parentElement.clientWidth;
  const H = Math.min(320, W * 0.55);
  canvas.width  = W; canvas.height = H;
  const ctx = canvas.getContext('2d');
  const dark = window.matchMedia('(prefers-color-scheme:dark)').matches;
  const fg = dark ? '#b4aaa0' : '#4a4540';
  ctx.clearRect(0, 0, W, H);
  const maxKb = Math.max(...data.map(d => d.kb));
  const bh = (H - pad.t - pad.b) / data.length;
  data.forEach((d, i) => {
    const y0 = pad.t + i * bh;
    const bw = (d.kb / maxKb) * (W - pad.l - pad.r);
    ctx.fillStyle = d.color;
    ctx.fillRect(pad.l, y0 + 4, bw, bh - 8);
    ctx.fillStyle = fg; ctx.font = '12px system-ui';
    ctx.textAlign = 'right';
    ctx.fillText(d.label, pad.l - 8, y0 + bh / 2 + 4);
    ctx.textAlign = 'left';
    ctx.fillText(d.kb + ' KB', pad.l + bw + 6, y0 + bh / 2 + 4);
  });
  ctx.fillStyle = fg; ctx.font = 'bold 13px system-ui'; ctx.textAlign = 'center';
  ctx.fillText('← Gzipped transfer size (KB) →', pad.l + (W - pad.l - pad.r) / 2, H - 8);
}
draw();
new ResizeObserver(draw).observe(document.body);
</script>
</body>
</html>

What makes this example correct

Zero external dependencies. Chart title and axis label present. Responsive via ResizeObserver. Dark mode handled by reading matchMedia at draw time. Alt text on the canvas element. Real data (not placeholder values). Every color has meaning (red = largest, green = smallest). The label text "← Gzipped transfer size (KB) →" tells the reader what the axis measures and its direction.
